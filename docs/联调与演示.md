# 联调与演示（开发计划阶段八）

本文档说明如何把各组件串成完整系统、演示流程脚本、故障排查，以及对照验收清单的就绪情况。

---

## 1. 两种运行形态

| 形态 | 命令 | 运行位置 | 说明 |
| --- | --- | --- | --- |
| **PC 演示** | `bash scripts/start_demo.sh` | 开发 PC | mock 模式起 Web，浏览器看完整系统；识别/抓取/库存全真实流转，硬件为 Mock |
| **真机部署** | `bash scripts/start_all.sh` | Atlas 开发板 | 编排 ROS2 服务 + 识别主程序 + Web，连接真实机械臂 |

命令行端到端演示（PC，不起 Web）：`python scripts/demo_full_system.py`

---

## 2. 真机启动顺序与依赖

`start_all.sh` 按下列顺序拉起，**顺序不能乱**（后者依赖前者）：

```
[1] ROS2 运动学服务 dofbot_server   ← 提供逆解，识别主程序依赖它
        │ (等待 ~5s 初始化)
        ▼
[2] 识别抓取主程序 run_power_sorting ← 调逆解 + 控制机械臂
        │
        ▼
[3] Web 可视化后端 uvicorn app:app  ← 展示状态/库存（与主程序共享状态）
```

依赖前置（缺一不可）：
- 已按 `scripts/setup_env.sh` 配好 ROS2/ais_bench 环境，`source setenv.sh`
- 相机已标定，`config/` 下有 `dp.bin` / `XYT_config.txt` / `offset.txt`
- `models/power_objects.om` 已转换并就位
- `ros2_ws/src/power_arm_control` 已实现 `SerialArmDriver` 与 `Ros2Kinematics`

日志在 `logs/`：`ros2_server.log` / `power_sorting.log` / `web.log`。

## 3. 演示流程脚本（答辩时照念）

建议顺序（对应开发计划阶段八建议演示流程）：

1. **展示硬件与系统首页**
   - 展示开发板、机械臂、摄像头、标定板连接。
   - 浏览器打开 `http://<开发板IP>:8000/`，介绍界面分区。

2. **色块识别与堆叠**（验收项 4）
   - 放置色块，点「开始识别抓取」。
   - 讲解状态流转：等待识别 → 已识别 → 抓取中 → 放置中 → 完成。
   - 演示连续堆叠多层。

3. **电力物品识别抓取**（验收项 5）
   - 放置电阻/扳手/线夹等，演示识别框 + 类别 + 置信度。
   - 机械臂按类别抓取并移动到对应库区（电子元器件区/工具区/电力金具区）。

4. **可视化界面联动**（验收项 6）
   - 页面实时显示识别类别、置信度、机械臂状态。
   - 库房统计随抓取增长，操作日志逐条记录。

5. **复位**
   - 点「回到初始姿态」，机械臂归位。

> PC 预演：上述 2-4 步可先用 `start_demo.sh` 在 PC 上跑一遍核对界面联动，再上真机。

## 4. 故障排查

| 现象 | 可能原因 | 处理 |
| --- | --- | --- |
| 识别主程序报「无法加载逆解客户端」 | dofbot_server 未启动 / power_arm_control 未实现 | 先确认 [1] 起来；检查 ros2_server.log |
| 抓取位置偏移 | 标定不准 / offset 未调 | 重新校准，调 `config/offset.txt`（见 config/calibration.md 附录 A） |
| 识别框不稳定 | 光照变化 | 固定光源；增加不同光照训练数据 |
| 逆解失败、机械臂不动 | 目标超出可达范围 | 限制可抓取区域，调 `config/arm.yaml` reach_bounds |
| 夹爪抓不稳、物体滑落 | 夹爪角/下降高度不当 | 调 `config/grasp.yaml` 对应类别参数 |
| Web 状态不更新 | 后台 runner 未启动 | 点「开始识别抓取」；检查 web.log |
| Web 端口被占用 | 端口冲突 | `start_all.sh <其它端口>` |
